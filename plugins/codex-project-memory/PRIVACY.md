# Privacy

Codex Project Memory is designed as a project-local Codex plugin. The core
runtime does not require a cloud service, embeddings, a vector database, or an
LLM call.

## Data Processed

The plugin scans the current repository and stores compact structural metadata:

- Relative POSIX file paths.
- File hashes and language/test/generated classifications.
- TypeScript/JavaScript symbols, imports, exports, module ownership, NestJS
  route metadata, and test adjacency.
- Parse/index warnings.
- Retrieval logs and duplicate candidates generated by deterministic local
  rules.
- Deterministic SVG/map/snapshot metadata generated from the local memory DB.

The plugin is designed to avoid public source dumps, absolute paths, Windows
backslashes in public outputs, and out-of-schema payloads.

## Storage

Project memory is stored inside the repository at:

```text
.codex/memory/
```

The SQLite database is:

```text
.codex/memory/memory.db
```

Generated frames and maps are also stored under `.codex/memory/`.

## Network Behavior

The core CLI and MCP server operate locally. They do not send repository memory
to a remote service by themselves.

Codex itself may still decide to call installed tools or services according to
the user's Codex configuration, permissions, and active prompt. This plugin's
MCP server communicates over local stdio.

## Retention And Deletion

The user controls retention by deleting the project memory folder:

```bash
rm -rf .codex/memory
```

On Windows PowerShell:

```powershell
Remove-Item -Recurse -Force .codex/memory
```

The next `pmem init` or `memory.refresh` can recreate the memory from the
current repository state.

## Secrets

Do not intentionally store secrets in repository files or project memory.
Generated public outputs are tested to avoid absolute paths and large source
dumps, but users should still review repository content and generated artifacts
before sharing them outside their organization.
